# Broski v0.5.2 Security Hardening Report

Date: March 6, 2026
Release: `v0.5.2`
Scope: security blockers + runtime/cache hardening

## Executive Summary

`v0.5.2` closes the highest-impact pre-release security gaps discovered in the v0.5 review cycle:

- Windows default-shell injection surface (`cmd.exe /C`) removed.
- Interactive redaction path moved from buffered collection to streaming.
- Strict Linux isolation no longer mounts host `/` wholesale.
- Secret manifest entries now use keyed hashing with workspace-local salt.
- Dynamic variable command output now has explicit bounded capture.

Additional hardening included:

- Runtime lock identity now combines PID liveness with Linux process-start identity.
- Cache restore validates artifact paths and object hashes before writing to workspace.

## Findings and Fixes

### 1. Windows command injection risk (critical)

Previous behavior:

- Default shell resolution on Windows could fall back to `cmd.exe /C`.
- Existing escaping strategy was POSIX-oriented and not safe for `cmd.exe` parsing.

Fix in `v0.5.2`:

- Removed default `cmd.exe` fallback.
- Default Windows shell execution now writes command content to temp `.ps1` script and runs:
  - `pwsh -ExecutionPolicy Bypass -NoProfile -File <script>`
  - fallback: `powershell -ExecutionPolicy Bypass -NoProfile -File <script>`
- Explicit shell overrides are still honored when intentionally configured.

### 2. Interactive redaction deadlock / OOM path (critical)

Previous behavior:

- Interactive tasks with redaction used `command.output()`.
- This buffered full process output and delayed terminal output until process exit.

Fix in `v0.5.2`:

- Uses `spawn()` for interactive redaction path.
- Reads stdout/stderr concurrently in dedicated threads.
- Applies line-by-line redaction and flushes immediately to terminal.
- Preserves interactive behavior while keeping redaction active.

### 3. Strict isolation confidentiality gap (critical)

Previous behavior:

- `bwrap` strict mode used `--ro-bind / /`, allowing read access across host filesystem.

Fix in `v0.5.2`:

- Replaced root bind with explicit base mounts (`/usr`, `/bin`, `/lib`, optional `/lib64`).
- Added isolated `/etc` handling for resolver config.
- Added tmpfs masking over host `$HOME`.
- Stage workspace remains explicitly bound and execution `chdir` remains in stage workspace.

### 4. Unsalted secret fingerprint material (critical)

Previous behavior:

- `@secret_env` values were hashed with unsalted fast BLAKE3 and persisted in manifest metadata.

Fix in `v0.5.2`:

- Introduced workspace-local 32-byte salt at `.broski/config/salt`.
- `@secret_env` manifest values now use keyed BLAKE3 with this salt.
- Fingerprinting interface updated to pass `FingerprintOptions` including `secret_env_key`.

### 5. Dynamic variable output memory/pipe-pressure risk (major)

Previous behavior:

- Dynamic variable command output was read unbounded after exit.
- Large outputs could trigger pipe-pressure stalls and timeout behavior.

Fix in `v0.5.2`:

- Added bounded capture at 1 MiB per stream.
- Added bounded draining logic to avoid unbounded memory while still draining pipes.
- Returns explicit parse error when output exceeds limit.

### 6. Runtime lock PID reuse resilience (major)

Previous behavior:

- Active lock checks relied on PID liveness only.

Fix in `v0.5.2`:

- Active lock payload now carries optional Linux process-start ticks.
- Lock is treated active only when PID is alive and start identity matches (when available).

### 7. Cache restore path boundary and object hash validation (major)

Previous behavior:

- Cache restore used relative paths from metadata without strict normalization checks.

Fix in `v0.5.2`:

- Added strict relative-path normalization (`no absolute`, `no ..`, non-empty normalized path).
- Added object hash validation (64 hex chars) before lookup.
- Workspace destination remains bounded to workspace root.

## Make vs Just vs Broski (Security and Integrity Perspective)

| Axis | Make | Just | Broski v0.5.2 |
| --- | --- | --- | --- |
| Deterministic content fingerprinting | No | No | Yes |
| Transactional output promotion | No | No | Yes |
| Cache explainability | No | No | Yes (`--explain`) |
| Secret-aware output redaction | No | Limited/manual | Yes |
| Default isolation model | Host-pass-through | Host-pass-through | Explicit mode (`off`/`best_effort`/`strict`) |
| Built-in dependency graph scheduling | Partial/manual | Limited | Yes |

Interpretation:

- Make and Just keep lower engine complexity but leave correctness/security controls mostly to user scripts.
- Broski has higher engine responsibility and therefore higher implementation burden, but offers stronger deterministic and security guarantees when hardened.

## Residual Risk and Follow-ups

- Strict mode remains Linux-only by design in current release line.
- PowerShell remains a runtime dependency for default Windows shell execution.
- `@secret_env` short values are intentionally excluded from redaction patterns to avoid log over-redaction side effects.

Recommended next hardening pass:

- Expand platform-specific runtime tests (especially Windows behavior coverage in CI).
- Continue focused audit of command/path handling in future feature additions.
- Re-run release gate on every patch release before public announcements.
